# Phase 41: Project Foundation - Research

**Researched:** 2026-02-28
**Domain:** NestJS 11 + Prisma 5 + PostgreSQL + Docker Compose — standalone repo scaffold
**Confidence:** HIGH

## Summary

Phase 41 creates a standalone `ai-sdr` repository (separate from the portfolio monorepo). The stack is NestJS 11 (TypeScript), Prisma 5.x ORM, and PostgreSQL 16 via Docker Compose. This is enabling infrastructure only — no user-visible behavior until Phase 43+. The four success criteria are: `docker compose up` starts both services and NestJS connects to Postgres; `prisma migrate dev` applies the Lead/AIOutput/EmailSequence/DemoLead schema; `ANTHROPIC_API_KEY` is readable via ConfigService and absent from client-side code and git history; `GET /health` returns 200.

The project's existing monorepo (phases 1-40) uses Prisma 5.22, NestJS 11, `postgres:16-alpine`, and the exact patterns documented here. The `ai-sdr` repo is standalone — no Turborepo, no workspace packages — making setup simpler than the monorepo. Docker Compose patterns used for devcollab in this project are the exact reference to follow: postgres service with `pg_isready` healthcheck, app service with `depends_on: condition: service_healthy`, migration as a separate step before the API starts.

Prisma 7 is the current latest release (Rust-free, 90% smaller bundle, output field required) but the project runs Prisma 5 in the monorepo. Use Prisma 5 for consistency and known-good patterns. Prisma 5 uses `provider = "prisma-client-js"` and the familiar `PrismaClient` import from `@prisma/client`. The NestJS ConfigModule with `isGlobal: true` is the correct pattern for `ANTHROPIC_API_KEY` — it is readable from any service via `ConfigService.get()` without re-importing the module.

**Primary recommendation:** Scaffold a standalone NestJS 11 repo with the existing project's Docker Compose and Prisma 5 patterns — do not attempt Prisma 7 migration, do not use monorepo tooling, enforce `.env` in `.gitignore` from commit zero.

### Pattern 1: PrismaService (existing project pattern)
**What:** Injectable service that wraps PrismaClient, connects on module init, disconnects on destroy.
**When to use:** Always — this is the standard NestJS Prisma integration pattern.
**Example:**
```typescript
// Source: Verified against /apps/devcollab-api/src/core/database/prisma.service.ts
import { Injectable, OnModuleInit, OnModuleDestroy } from '@nestjs/common';
import { PrismaClient } from '@prisma/client';

@Injectable()
export class PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy {
  async onModuleInit() {
    await this.$connect();
  }

  async onModuleDestroy() {
    await this.$disconnect();
  }
}
```

Note: In the standalone repo, `PrismaService` extends `PrismaClient` directly (unlike devcollab-api which wraps a shared package client). This is the cleaner pattern for a standalone repo.

### Pattern 2: ConfigModule with Global + Validation
**What:** ConfigModule loaded globally so any service can inject ConfigService without re-importing.
**When to use:** Always — ensures ANTHROPIC_API_KEY is validated at startup and accessible everywhere.
**Example:**
```typescript
// Source: NestJS docs + devcollab-api app.module.ts reference
import { Module } from '@nestjs/common';
import { ConfigModule } from '@nestjs/config';
import * as Joi from 'joi';
import { DatabaseModule } from './database/database.module';
import { HealthModule } from './health/health.module';

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      validationSchema: Joi.object({
        NODE_ENV: Joi.string().valid('development', 'production', 'test').default('development'),
        PORT: Joi.number().default(3001),
        DATABASE_URL: Joi.string().required(),
        ANTHROPIC_API_KEY: Joi.string().required(),
      }),
    }),
    DatabaseModule,
    HealthModule,
  ],
})
export class AppModule {}
```

### Pattern 3: Docker Compose with health-gated startup
**What:** Postgres with `pg_isready` healthcheck; NestJS app depends on `service_healthy`.
**When to use:** Always in Docker Compose — prevents NestJS from starting before Postgres accepts connections.
**Example:**
```yaml
# Source: Verified against /home/doctor/fernandomillan/docker-compose.yml (devcollab pattern)
version: '3.8'

services:
  postgres:
    image: postgres:16-alpine
    container_name: ai-sdr-postgres
    restart: unless-stopped
    ports:
      - '5436:5432'
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-aisdr}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-aisdr}
      POSTGRES_DB: ${POSTGRES_DB:-aisdr}
    volumes:
      - ai-sdr-pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ['CMD-SHELL', 'pg_isready -U ${POSTGRES_USER:-aisdr}']
      interval: 10s
      timeout: 5s
      retries: 5

  api:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: ai-sdr-api
    restart: unless-stopped
    ports:
      - '3001:3001'
    depends_on:
      postgres:
        condition: service_healthy
    env_file:
      - .env
    environment:
      NODE_ENV: production
      DATABASE_URL: postgresql://${POSTGRES_USER:-aisdr}:${POSTGRES_PASSWORD:-aisdr}@postgres:5432/${POSTGRES_DB:-aisdr}

volumes:
  ai-sdr-pgdata:
```

### Anti-Patterns to Avoid
- **ANTHROPIC_API_KEY in any file committed to git:** Even in a `config` file — use `.env` only, verified `.gitignore` from day one.
- **Prisma 7 on first attempt:** Prisma 7 requires `output` field in generator, driver adapter pattern, and different import path. The monorepo runs Prisma 5 — use 5 for this standalone repo too.
- **Starting NestJS before Postgres is ready:** Never use `depends_on` without `condition: service_healthy`. `depends_on` alone only waits for container start, not Postgres readiness.
- **Running `prisma migrate dev` in Docker CMD:** Use `prisma migrate deploy` in Docker (idempotent, safe for production). Use `prisma migrate dev` only locally for schema development.
- **Global DATABASE_URL without service name:** In Docker Compose, NestJS must use the service name (`postgres`) not `localhost` in the connection string. `localhost` won't resolve to the Postgres container.
- **Missing `.gitignore` on repo init:** `nest new` does not always add `.env` to `.gitignore`. Add it manually before first commit.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Environment variable validation | Manual `if (!process.env.X) throw` checks | `@nestjs/config` + `joi` validationSchema | Joi validates type, required fields, and allowed values at startup before any request |
| Database connection lifecycle | Manual connect/disconnect in controllers | `PrismaService` with `OnModuleInit`/`OnModuleDestroy` | NestJS lifecycle hooks guarantee connect before first request and disconnect on graceful shutdown |
| Docker Postgres readiness | `sleep 5` in CMD | `healthcheck` + `depends_on: condition: service_healthy` | Sleep is flaky; `pg_isready` is deterministic |
| Health check routing | Manual Express route | `HealthModule` with NestJS `@Controller` | Integrates with NestJS module system, no bypassing DI |

**Key insight:** The existing devcollab-api in this repo already uses all these patterns correctly. Copy the patterns, not custom solutions.

## Common Pitfalls

### Pitfall 1: ANTHROPIC_API_KEY leaks into git history
**What goes wrong:** Developer adds `.env` to the first commit before `.gitignore`, or hardcodes the key in a config file. Once in git history, removing it requires `git filter-repo` and force-push.
**Why it happens:** `nest new` scaffolds quickly and `.env` may not be in the auto-generated `.gitignore`.
**How to avoid:** Step one of repo init is create `.gitignore` with `.env`. Verify with `git status` before first commit.
**Warning signs:** `git status` shows `.env` as untracked but listed as "Changes to be committed."

### Pitfall 2: DATABASE_URL points to localhost inside Docker
**What goes wrong:** NestJS container starts, tries to connect to `localhost:5432`, fails because Postgres is in a different container.
**Why it happens:** `.env` has `DATABASE_URL=postgresql://...@localhost:5432/...` which works locally but not in Docker Compose.
**How to avoid:** In Docker Compose, set `DATABASE_URL` in the `environment` block using the service name: `@postgres:5432`. The `.env` file uses `localhost` for local dev; Docker Compose overrides it via the `environment` block.
**Warning signs:** NestJS exits on startup with `Can't reach database server at localhost:5432`.

### Pitfall 3: `prisma migrate dev` vs `prisma migrate deploy`
**What goes wrong:** Using `migrate dev` in Docker CMD causes interactive prompts in CI and fails non-interactively. Using `migrate deploy` locally may skip shadow database creation and miss drift.
**Why it happens:** `migrate dev` is designed for local development; `migrate deploy` is designed for CI/production.
**How to avoid:** Use `migrate dev` locally to create migration files. Use `migrate deploy` in Docker CMD and CI.
**Warning signs:** Docker CMD hangs waiting for input, or exits with "shadow database" errors.

### Pitfall 4: NestJS 11 Express wildcard route syntax changed
**What goes wrong:** Wildcard routes like `@Get('*')` do not work in NestJS 11 with Express v5.
**Why it happens:** Express v5 requires named wildcards: `@Get('*splat')`.
**How to avoid:** Phase 41 doesn't need wildcard routes, but document for later phases.
**Warning signs:** Route not matched, 404 on expected wildcard paths.

### Pitfall 5: Prisma client not generated before build
**What goes wrong:** TypeScript build fails with "Cannot find module '@prisma/client'" or missing type errors.
**Why it happens:** `@prisma/client` types are generated from `schema.prisma` on `prisma generate`. Without running generate first, types don't exist.
**How to avoid:** Add `prisma generate` as a `postinstall` script in `package.json`, and run it explicitly in Dockerfile before `nest build`.
**Warning signs:** `npx tsc` fails with type errors pointing to `@prisma/client`.

### Pitfall 6: Port conflicts with existing monorepo services
**What goes wrong:** `docker compose up` fails because ports 5432, 3000 are already in use by teamflow-postgres, devcollab-postgres, or other monorepo services.
**Why it happens:** The monorepo already maps ports 5434 (teamflow) and 5435 (devcollab). The new ai-sdr Postgres should use a distinct port.
**How to avoid:** Use port 5436 for ai-sdr Postgres (continues the monorepo's port convention), and port 3001 for the NestJS API.
**Warning signs:** `docker compose up` fails with "address already in use."

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `depends_on: [postgres]` (no condition) | `depends_on: postgres: condition: service_healthy` | Docker Compose v3.x+ | Eliminates race condition; NestJS waits for Postgres to accept connections |
| `prisma-client-js` provider → node_modules output | `prisma-client` provider → src/generated output (Prisma 7) | Prisma 7.0 (2025) | This project stays on Prisma 5 for monorepo consistency |
| `enableShutdownHooks(app)` on PrismaService | `app.enableShutdownHooks()` in main.ts | NestJS ~9+ | Simpler; PrismaService only needs OnModuleDestroy |
| NestJS 10 | NestJS 11 (Express v5, Fastify v5, JSON logger) | NestJS 11 (2025) | Wildcard routes need `*splat` syntax; JSON log built-in |

**Deprecated/outdated:**
- `version: '3.8'` in docker-compose.yml: Docker Compose v2 ignores this field; harmless to include for compatibility but not required.
- `PrismaService.enableShutdownHooks(app)`: Old pattern; use `OnModuleDestroy.$disconnect()` instead.

## Open Questions

1. **Prisma 5 vs Prisma 7 for standalone repo**
   - What we know: Monorepo uses Prisma 5.22. Prisma 7 has a smaller bundle, faster queries, but requires breaking changes (output field required, adapter pattern, different import paths).
   - What's unclear: Whether using Prisma 7 in the standalone repo while Prisma 5 is in the monorepo creates confusion.
   - Recommendation: Use Prisma 5 (`^5.22.0`) for the standalone repo. Consistency with known patterns is more valuable than Prisma 7 bundle savings for a portfolio project.

2. **Port assignment for ai-sdr services**
   - What we know: teamflow-postgres uses 5434, devcollab-postgres uses 5435. API ports: devcollab-api uses 3003.
   - What's unclear: Whether ai-sdr will run alongside the monorepo or in isolation.
   - Recommendation: Use port 5436 for ai-sdr Postgres, port 3001 for ai-sdr API. These don't conflict with any existing monorepo services.

3. **DemoLead schema design**
   - What we know: ROADMAP.md says DemoLead is a model. Phase 46 references "6-8 fictional leads" with `faker.seed(42)` and idempotent upserts.
   - What's unclear: Whether DemoLead should have all the same fields as Lead (duplicating schema) or just seed metadata.
   - Recommendation: DemoLead is a seed-control table only — stores `seedKey` (faker seed identifier) for idempotent re-seeding. Actual seeded data should be Lead records, not DemoLead records. Keep DemoLead minimal.
